# 小说创作编排工作流

> 编排 chinese-novelist（引擎）+ chinese-writing-skills（知识库）= 端到端的小说创作系统

## 执行入口

当用户说"开始写小说""创作一部小说""帮我写一部XX小说"时，按以下流程启动：

---

### 第 0 步（超长篇特有）：弧级规划 + 首卷初始化

当检测到用户在创作 ≥300 章的小说时，详见 references/ultra-long-novel-design.md。

#### 0.A 弧级规划（v3.2 增强版）

1. 告知用户将使用超长篇模式
2. 与用户共同定义 6-10 个弧（每个弧 300-500 章）
3. 每个弧定义以下五个维度：

```
弧N「名称」（第X-Y章）

1. 叙事骨架：
   - 起点状态：主角/世界在弧开始时的状态
   - 终点状态：主角/世界在弧结束时的状态
   - 核心冲突：本弧的主要矛盾
   - 关键转折点：2-3个重大事件

2. 情感主线（⭐新增 — 编剧视角）：
   - 读者的情感旅程：从[起始情绪] → [中间情绪] → [终点情绪]
     例：从"好奇+期待"→"不安+紧张"→"震撼+释然"
   - 主角的情感弧线：从[起始] → [中间] → [终点]
     例：从"恐惧强者"→"愤怒"→"理解力量的本质是保护"

3. 线交织设计（⭐新增 — 编剧视角）：
   - 本弧涉及的叙事线：列出所有在本弧活跃的线
   - 关键交织点：
     | 章 | 交汇线 | 交汇效果 |
     |:---:|-------|---------|
     | C020 | 主线×支线B | 主角从王老那里得到第一条线索 |
     | C120 | 主线×支线D | 魔教事件迫使主角做出道德选择 |
   - 交织的戏剧原则：交汇不能只是偶遇，每次交汇必须推动至少一条线
     的状态变化

4. 三级高潮地图（⭐新增 — 编剧视角）：
   - 小高潮（每50-80章）：
     | 章 | 类型 | 目标读者反应 |
     |:---:|------|------------|
     | C050 | 战斗小高潮 | "主角变强了！" |
     | C120 | 情感小高潮 | "王老居然..." |
   
   - 中高潮（每150章左右）：
     | 章 | 类型 | 目标读者反应 |
     |:---:|------|------------|
     | C180 | 复仇中高潮 | "终于！但怎么这么空虚" |
   
   - 大高潮（弧末尾）：
     | 章 | 类型 | 目标读者反应 |
     |:---:|------|------------|
     | C0500 | 弧终大高潮 | "这段旅程值了" |

5. 读者体验设计：
   - 本弧的核心阅读体验：如「压迫→愤怒→压抑→爆发→满足」
   - 本弧结束时读者应该有的感觉：[一句话]
```

4. 生成 `arc-plan.md`（包含上述五个维度），用户确认

#### 0.A2 从弧规划中提取故事线 + 分篇章

在弧规划确认后、首卷初始化前，必须完成：

1. **拆篇章**：从弧1（300-500章）拆出3-4个篇章（100-150章/篇章）
   - 每个篇章有独立的主题、起点、终点
   - 生成 `part-plan.md`（弧1所有篇章的规划）

2. **识别故事线**：从弧规划的所有内容中提取叙事线
   - 每条线有独立的目标→达成轨迹
   - 识别规则：独立目标 + 跨越≥20章的轨迹 = 一条故事线
   - 为每条线创建 `storylines/{线名}.md` 初始文件
   - 初始文件包含：触发章、第一段的规划、情绪弧预设

3. **生成故事线总览**：`storylines-overview.md`
   - 所有线的ID、优先级、类型、预期章范围

#### 0.B 首卷初始化

1. 选择弧1的前 50 章作为第一卷
2. 生成详细卷规划 `volume-plans/vol-01-XXX.md`
3. 初始化全部项目文件（v3 全量）：
   - `arc-plan.md` — 弧级规划（已生成）
   - `arc-summaries/` — 弧完成时填充
   - `part-plan.md` — 篇章规划（从弧1拆3-4个篇章）⭐v3
   - `part-summaries/` — 篇章完成时填充 ⭐v3
   - `volume-plans/` — 卷规划目录
   - `volume-summaries/` — 卷总结目录
   - `storylines/` — 每条叙事线一个文件 ⭐v3
   - `storylines-overview.md` — 所有故事线总览表 ⭐v3
   - `part-chapter-map.json` — 篇章-章节映射+呼应表 ⭐v3
   - `chapters/` — 按卷分子目录
   - `characters/` — 每个活跃角色一个状态机文件
   - `relation-matrix.md` — 人物关系矩阵
   - `foreshadowing.json` — 伏笔追踪
     - Schema: id, description, chapter_planted, chapter_revealed,
       expected_reveal_range[min,max], status(planted/revealed/abandoned),
       type(real/accidental/fake), depends_on[id], priority(P0/P1/P2),
       final_deadline(超时废弃期限), abandon_reason
     - Schema: id, description, chapter_planted, chapter_revealed,
       expected_reveal_range[min,max], status(planted/revealed/abandoned),
       type(real/accidental/fake), depends_on[id], priority(P0/P1/P2),
       final_deadline(超时废弃期限), abandon_reason
   - `world-state.md` — 世界状态快照
   - `style-baseline.json` — 风格基线（写完30章后校准）
   - `creative-notes.md` — 创作决策笔记
   - `chapter-index.json` — 章节结构化索引
     - 字段含 world_time_start/world_time_end (世界时间戳，
       格式YYYY-MM-DD或"第X纪元YY年")，用于时间一致性追踪 (v3.2)
   - `notable-lines.json` — 金句与名场景索引
     - 金句(golden_lines): 章节/文本/说话人/类型/标签
     - 名场景(memorable_scenes): 章节/场景名/字范围/情绪/画面描述/标签
       (v3.2: 独立于金句的名场景追踪——画面感记忆)
   - `session-log.jsonl` — 写作会话日志
   - `quality-reports/` — 卷级+篇章级+弧级审查报告
   - `global-preference.md` — 全局基调偏好
   - `writing-plan.json` — 当前卷的调度状态
4. **写完第1章后暂停，让用户确认基调**

### 永久设定索引 `immutable-rules.md`（超长篇初始化时生成）

> **审核员视角**：最多烂尾来自作者忘了自己的早期设定。
> 第1章写"灵气稀薄修炼极难"，第1500章变成"天灵根遍地走" → 读者差评。
> 此文件记录永远不变的铁则，每次创作前对照检查。

**在首卷初始化时创建 `immutable-rules.md`**，包含：

```markdown
# 永久设定索引
最后更新：第 100 章

## 世界观铁则（不可违背，除非专门写逆转章节）

### 物理/魔法法则
1. 灵气稀薄：大陆平均灵气浓度 ≤ 3级（1-10级标准）
2. 修炼体系仅三种：气修、体修、魂修。不存在第四种。
3. 死亡不可逆转。任何"复活"都是幻觉/假死/夺舍。
4. 传送阵仅存在于八大主城之间，距离限制 ≤ 3000里。
5. ...

### 社会法则
1. 宗门等级制：外门→内门→核心→长老→掌门。不可跳跃。
2. 散修地位低于宗门弟子，但不受宗门规则约束。
3. ...

## 角色永久设定（不可违背，除非角色经历了明确标注的转变）

### 张三（主角）
- 出身：孤儿，父母死于[不剧透]
- 天赋：中等偏上，不是天才。靠努力而非天赋成长。⚠️ 不可写成"万年一遇的天才"
- 性格底线：不杀无辜。⚠️ 如果张三要打破这条底线，需要一个完整的弧来解释。
- 已知信息限制：第1-500章期间不知道生父身份 ⚠️ 不能提前写他知道

### 李四
- 本质：外表冷酷、内心重情。⚠️ 永远不能写成真正的冷漠
- 曾背叛主角一次（第200章），但当时是被胁迫的 ⚠️ 不能写成主动背叛
- ...

## 历史铁则（已经发生，不可改变）

1. 500年前的神魔大战：人类惨胜，魔尊被封印。封印位置未知（第300章揭示）。
2. 天剑宗是500年前大战后建立的第一个宗门。
3. ...

## 命名规则
- 宗门名：双字 + "宗/门/阁"（如天剑宗、苍云门）
- 功法名：四字（如《天剑九式》《苍云心经》）
- 地名：单字州名 + 城市名（如青州·青云城）
```

**使用规则**：
- 每次创作前（在上下文装载清单中），对比本章大纲和 `immutable-rules.md`
- 如果大纲有违背铁则的嫌疑 → 调整大纲
- 如果确实需要打破某条铁则 → 必须在创作笔记中记录「第X章打破铁则Y」
- 卷末审查时检查本卷是否有未经记录的设定违背

### 续写预热协议（每次恢复创作时强制执行）

无论是中断恢复、跨会话继续、还是新卷开始，**必须先执行预热 SOP**。

**⚠️ 预热前置步骤：先确认进度**
在读取任何文件之前，先检查 `writing-plan.json` 或 `session-log.jsonl` 最新记录，
确认下一次创作的章号、卷号、篇章。如果是从卷末尾恢复，确认是新卷的规划还是
继续检查。**不要在没有确认章号的情况下加载预热文件。**

```
续写预热 14 步：
 0. 确认下一章号 → 读 writing-plan.json 或 session-log.jsonl
1. 读 arc-plan.md → 了解最终目标
2. 读 arc-summaries/当前弧 → 本弧目标
3. 读 volume-summaries/上一卷 → 最近发生了什么
4. 读 volume-summaries/当前卷 → 现在在哪
5. 读 creative-notes.md 最近20条 → 创作决策
6. 读 storylines-overview.md → 所有活跃故事线状态 ⭐v3
7. 读本章涉及的 storylines/{线名}.md → 故事线完整叙事路径 ⭐v3
8. 读 part-chapter-map.json → 本章的呼应关系和读者情绪目标 ⭐v3
9. 读上章全文（5000字）→ 文字衔接感
10. 读 relation-matrix.md → 人物关系直觉
11. 读 world-state.md → 世界观状态
12. 读 style-baseline.json → 风格锚点
13. 如果最近50章内跨了篇章边界 → 重读上一篇章的 part-summary ⭐v3
14. 向用户输出 500 字预热摘要，确认理解正确
```

预热摘要模板见 references/ultra-long-novel-design.md §3.2。

### 第 2 步：规划阶段 — 模式分支

#### 短中篇/长篇模式（≤299章）

在 novelist 生成大纲和人物档案后，进入规划确认前，**自动挂载**对应的技法模块进行深度注入。

#### 2.1 读取技法模块

从 chinese-writing-skills 中读取以下模块，为规划提供技法支撑：

```
□ 01-人物塑造 → 检查人物档案是否有弧光、矛盾、侧面揭示
□ 02-情节设计 → 检查大纲冲突层次、节奏曲线、高潮布局
□ 08-爽点设计 → 检查爽点分布和密度（确保 writer JSON 有合理的爽点章标注）
□ 09-开篇技法 → 确认前 3 章标记为 opening，符合黄金三章标准
□ 07-悬念构建 → 检查总悬念、阶段悬念、章节悬念是否分层设置
□ 10-结尾技法 → 确认最后 1-2 章标记为 ending，结局类型明确
```

#### 2.1.5 验证 chapterType 标注

**在确认规划前，必须验证 novelist 生成的写作计划 JSON 中每章的 `chapterType` 标注是否合理**：

```
□ 第1-3章 chapterType 是否为 "opening"（如果没有，要求 novelist 修正）
□ 爽点章是否标注为 "pleasure"（如果没有，检查是否漏标或爽点分布有问题）
□ 反转章是否标注为 "twist"
□ 高潮章是否标注为 "climax"
□ 最后 1-2 章是否标注为 "ending"
□ 过渡章是否标注为 "transition"
□ chapterType 分布是否合理（不能全是 setup 或全是 pleasure）
```

#### 2.2 技法对照检查

对 novelist 生成的规划内容执行对照检查：

```
人物档案检查：
  □ 主角是否有明确的弧光（开始→结束的变化）？
  □ 主角是否有至少一处内在矛盾？
  □ 反派是否有自己的逻辑？是否是主角的"黑暗镜像"？
  □ 配角是否每个都有明确功能？

情节大纲检查：
  □ 每章是否至少设计了一个冲突？
  □ 冲突是否有层次叠加（人vs人 + 人vs自我 + ...）？
  □ 高潮赌注是否最大化？
  □ 爽点分布是否合理（蓄力→小爽→压抑→中爽→压抑→大爽）？

开篇检查：
  □ 前500字是否有强钩子？
  □ 黄金三章结构是否完整？
  □ 是否有天气/起床/吃早餐等致命错误？
```

#### 2.3 生成增强规划

根据技法检查结果，对规划提出优化建议（3-5条），展示给用户选择：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 技法增强建议（根据知识库分析）

1. 🔶 人物增强：主角弧光不够清晰
   当前：侦探查案找到凶手
   建议：增加"侦探自己也做过类似的事"的内心挣扎
   依据：人物塑造模块 §反派是主角的黑暗镜像

2. 🔶 爽点优化：第三章打完脸，第五章才第二个爽点
   建议：第四章增加一个小爆点（配角反水？新线索颠覆认知？）
   依据：爽点设计模块 §密度控制 — 你的题材适合每2-3章一个爽点

3. ✅ 开篇通过：第一章从争执开始，符合黄金三章标准

你希望应用这些建议后再开始创作，还是直接用当前规划？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 第 2.5 步：超长篇模式 — 创作前对齐检查

## 卷完成仪式（超长篇 ≥300章）

每完成 50 章（一卷），强制执行：

```
1. 生成完整卷总结（1000-2000字）→ volume-summaries/
2. 更新所有活跃人物状态机 → characters/*-state.md
3. 风格漂移检测 → 对比 style-baseline.json
4. 审计伏笔健康度：
   - 回收率 ≥ 40%
   - 是否存在「超时伏笔」（chapters_since_planted > expected_reveal_range_max + 50）
     → 标记为「废弃」并记录原因
   - 逐条检查：每项伏笔是否有 chapter_planted 和 expected_reveal_range
   - 检查依赖关系：有 depends_on 字段的伏笔，其依赖是否已回收
5. 检查弧级规划对齐度
6. 🔴 跨文件一致性审计（v3新增 — 详见§跨文件一致性审计）
6.5 🟠 情节模式重复检测（v3.2新增 — 编剧+审核员联合建议）：
   - 对比本卷50章的冲突类型分布（人vs人/人vs自我/人vs社会/人vs自然）
     如果某一种类型占比 > 60% → 警告
   - 对比本卷各章的转折方式：
     如果「神秘人突然出现」「意外发现」「角色背叛」连续使用 >=3次 → 警告
   - 对比本卷各章的高潮模式：
     如果连续 >=5章用了同一种高潮模式 → 警告
   - 输出重复模式警告到审查报告
6.5 🟠 情节模式重复检测（v3.2新增 — 编剧+审核员联合建议）：
   - 对比本卷50章的冲突类型分布（人vs人/人vs自我/人vs社会/人vs自然）
     如果某一种类型占比 > 60% → 警告
   - 对比本卷各章的转折方式：
     如果「神秘人突然出现」「意外发现」「角色背叛」连续使用 >=3次 → 警告
   - 对比本卷各章的高潮模式：
     如果连续 >=5章用了同一种高潮模式 → 警告
   - 输出重复模式警告到审查报告
7. 生成卷末审查报告 → quality-reports/
8. 如果本卷跨了一个篇章边界 → 触发篇章完成仪式
9. 向用户展示摘要，确认后进入下一卷
```

### 跨文件一致性审计（每卷强制执行）

> **核心原则**：21个文件必须互相自洽。以下是必须检查的维度：

```
[卷末一致性审计清单]

□ 人物状态 ↔ 关系矩阵：
   - 对比 characters/*-state.md 中的"当前关系状态"字段
     和 relation-matrix.md 中的对应行/列
   - 如果 A-state.md 说"与B关系破裂"但矩阵说"A-B=信任"→ 修复

□ 人物状态 ↔ 最近5章正文：
   - 抽样最近5章，检查出场角色的行为是否与状态文件一致
   - 如果状态文件说"张三失明"但最近章他没表现出失明 → 警告

□ 故事线进度 ↔ 篇章规划：
   - storylines-overview.md 中每条线的"当前段"
     是否与 part-plan.md 中对应篇章的进度匹配
   - 如果有线落后或超前 → 调整规划

□ 伏笔文件 ↔ 章节索引：
   - foreshadowing.json 中的 chapter_planted/chapter_revealed
     是否与 chapter-index.json 中对应章的 key_events 一致
   - 如果伏笔说在第50章埋入但章节索引说第50章没写这个事件 → 修复

□ 世界状态 ↔ 最近5章正文：
   - world-state.md 描述的状态是否与近期章节一致
   - "帝国已覆灭"但最近章节写"帝国依然强盛" → 警告

□ 风格基线 ↔ 本卷章节：
   - style-baseline.json 的基线值 vs 本卷5个采样章的实际值
   - 偏差超过阈值的项标记到审查报告

□ 篇章-章节映射 ↔ 故事线总览：
   - part-chapter-map.json 的 storylines_active 字段
     是否与 storylines-overview.md 的活跃线一致

审计输出：在 quality-reports/vol-XX-review.md 中增加「一致性审计」小节
修复：产生不一致的条目自动标记为待修复，下一段创作前先修复
```

## 弧级重规划协议（v3.2 新增）

> **审核员视角**：超过500章的小说必定经历「推翻早期设定」。
> 如果弧规划只能加不能改，agent 会在错误的方向上越走越远。

### 触发条件
- 弧中任一篇章完成后，实际进展与规划偏差 > 30%
- 出现了规划中没有的重大叙事机会
- 早期设定被证明不可行
- 用户在弧中途提出方向性修改

### 协议流程

```
1. 偏差分析：
   - 对比当前的 arc-plan.md vs 实际进展
   - 列出所有偏差项及影响范围

2. 追溯影响分析：
   - 如果重规划，已写完的章中有哪些需要修订？
   - 正在进行的卷需要改什么？
   - 下一个篇章的规划需要如何调整？

3. 提案：
   - 生成重规划提案（不是全量重写，是修改受影响部分）
   - 标注：哪些已成章需要修改（轻量）、哪些需要重写（重量）
   - 展示给用户，确认后再执行

4. 执行：
   - 更新 arc-plan.md 的受影响部分
   - 更新 storylines-overview.md
   - 更新 part-plan.md（受影响篇章的规划）
   - 在 creative-notes.md 记录重规划原因和决策

5. 追溯修复（可选，用户决定）：
   - 如果用户同意修改已完成章节：
     使用「章节修改协议」逐章修改
```

## 章节修改协议（中途修改时使用）

> 写作不是直线流程。用户可能要求修改已完成章节、删除角色、重写场景。
> 以下协议确保修改不会破坏21个文件的一致性。

### 轻量级修改（改一段对话/一处措辞）

只改章节文件，**不更新其他文件**（除非修改了伏笔/关系/角色行为）

### 中度修改（改一个场景/一次互动）

```
1. 修改章节文件（替换对应段落）
2. 检查：修改是否涉及角色关系变化？
   → 是 → 更新 relation-matrix.md
3. 检查：修改是否涉及伏笔？
   → 是 → 更新 foreshadowing.json
4. 检查：修改是否改变了角色行为/状态？
   → 是 → 更新 characters/{角色名}-state.md
5. 追加修改日志到 session-log.jsonl
```

### 重度修改（删角色/重大剧情改动）

```
1. 确认波及范围：
   - 该角色在哪些章出场？（查 chapter-index.json）
   - 该角色涉及哪些故事线？（查 storylines-overview.md）
   - 该角色在哪些伏笔中？（查 foreshadowing.json）
2. 列出所有受影响文件
3. 逐文件修改（从最近章往前改，保持时间线一致）
4. 修改完成后执行「跨文件一致性审计」
5. 向用户展示修改影响范围，确认不破坏其他线
```
