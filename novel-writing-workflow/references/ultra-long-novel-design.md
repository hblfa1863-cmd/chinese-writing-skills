# 超长篇小说创作系统设计

> 面向 3000 章（1000+ 万字符）量级的小说创作系统架构

## 1. 核心矛盾

```
上下文窗口：~200K tokens
3000章的摘要累积：~135K tokens（仅摘要就占 67%）
3000章的伏笔累积：~60K tokens
3000章的人物状态：~50K tokens
────────────────────────
总信息量 >> 窗口容量
```

**传统做法（逐章摘要叠加）在 300 章就爆炸**。

## 2. 解决原则：分层记忆（Hierarchical Memory）

不是"把所有信息压缩放进上下文"，而是"在上下文里只保留导航地图，需要什么信息时去外部文件系统查找"。

### 记忆分层模型

```
                               ┌──────────────────────────┐
                               │  上下文（200K tokens）     │
                               │  ┌────────────────────┐  │
                               │  │ 当前章大纲         │  │
                               │  │ 上章完整摘要(300字)│  │
                               │  │ 当前卷总结(500字)  │  │
                               │  │ 当前弧总结(500字)  │  │
                               │  │ 总览摘要(300字)    │  │
                               │  │ 人物状态机(精简版) │  │
                               │  │ 活跃伏笔索引       │  │
                               │  │ 世界状态简表       │  │
                               │  └────────────────────┘  │
                               │          总计 < 5K tokens │
                               └──────────┬───────────────┘
                                          │ 需要时检索
          ┌───────────────┬───────────────┼───────────────┬───────────────┐
          ▼               ▼               ▼               ▼               ▼
   ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ 章节文件  │  │ 卷总结   │  │ 弧总结   │  │ 人物档案  │  │ 伏笔文件  │
   │ ch001-   │  │ vol-01-  │  │ arc-01-  │  │ 主角.     │  │ foresha-  │
   │ ch3000   │  │ summary  │  │ summary  │  │ state.md  │  │ dowing.md │
   │ .md      │  │ .md      │  │ .md      │  │           │  │           │
   └──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘
```

### 上下文占用分析（5K tokens 模型）

| 内容 | 大小 | 格式 |
|------|------|------|
| 当前章大纲（从规划中提取） | ~200 tokens | 核心事件+出场人物+悬念钩子 |
| 上章完整摘要 | ~300 tokens | 300字叙事摘要 |
| 当前卷总结 | ~500 tokens | 本卷(50章)的故事脉络 |
| 当前弧总结 | ~500 tokens | 本弧(3-5卷=150-250章)的大方向 |
| 总览摘要 | ~300 tokens | 整部小说的终极目标+当前进度 |
| 人物状态机（压缩版） | ~500 tokens | 当前卷所有活跃角色状态 |
| 活跃伏笔索引 | ~300 tokens | 未回收伏笔ID+预期回收章节 |
| 世界状态简表 | ~200 tokens | 关键势力/地点/规则的变化 |
| 上一章全文（仅当前创作时） | ~2000 tokens | 创作时参考 |
| **合计** | **~5K tokens** | 占上下文的 2.5% |

## 3. 三层规划体系

3000 章不能一次性规划。采用三段式降层规划：

### 3.1 弧级规划（Arc Plan）—— 最高层

> 一部 3000 章的小说分为 6-10 个大弧，每个弧 300-500 章
> 每个弧有独立的：主题、核心冲突、起点状态、终点状态、关键转折点

```
弧1: 觉醒之卷 (第1-300章)
  起点: 平凡少年
  终点: 获得力量但失去重要之人
  核心冲突: 弱小 vs 世界
  关键转折点: 第150章(力量觉醒), 第250章(背叛), 第300章(复仇之誓)

弧2: 征途之卷 (第301-600章)
  起点: 复仇之誓
  终点: 成为一方势力之主
  ...
```

### 3.2 卷级规划（Volume Plan）—— 中层

> 每个弧内部再分为 3-6 卷，每卷 50-100 章
> 卷是**创作调度单位**——写完一卷后重规划下一卷

```
卷1.1: 平凡之日 (第1-50章)
  本卷主题: 展示"平凡"的珍贵
  幕结构: 开端→触发事件→抉择→进入新世界
  每5章一个小节奏: 5章铺垫→5章升级→5章转折→5章高潮→5章回落
  爽点分配: 第15章(小), 第30章(中), 第48章(大)

卷1.2: 觉醒之痛 (第51-150章)
  ...
```

### 3.3 章级规划（Chapter Plan）—— 底层

> 与现有 novelist 兼容，但由卷规划驱动
> 卷启动时一次性规划 50 章的大纲（滚动规划每 10 章刷新后续 5 章）
> 每章标注 chapterType

### 3.4 卷间重规划机制

每完成一卷（50章），强制执行：

```
卷完成仪式：
1. 生成本卷总结（1000-2000字）
2. 更新人物状态机（所有活跃角色）
3. 检查弧级目标对齐度
4. 评估是否需要调整下一卷规划（最多允许调整下一卷的前25章）
5. 更新未回收伏笔清单
6. 用户确认后进入下一卷
```

## 4. 人物状态追踪系统

人物不能再像短篇那样用段落描述——需要用状态机。

### 4.1 人物状态文件结构

每个活跃角色一个状态文件 `characters/{角色名}-state.md`

```markdown
# 张三（主角）状态机
最后更新：第 347 章

## 身份状态
- 境界/实力：金丹中期 → 元婴初期（第287章突破）
- 身份：散修 → 天剑宗内门弟子（第120章）→ 核心弟子（第245章）
- 地位：无名 → 宗门新星 → 被长老忌惮

## 关系状态
- 李四（挚友）：信任→第203章猜疑→第298章和解→现在：并肩作战
- 王五（爱人）：第89章相遇→第200章表白→第347章订婚→现在
- 赵六（宿敌）：第156章结仇→第300章一战→重伤未死→下落不明
- 天剑宗：归属感建立中，但隐约怀疑宗门有秘密

## 心理状态
- 核心动机：保护身边人（非变强/变最强）
- 当前困惑：力量越大，失去越多，值得吗？
- 性格变化：谨慎→第156章后变得果断→第287章后略微自负
- 隐性创伤：第300章赵六说出的真相（关于父母）

## 能力与物品
- 武器：残剑（第45章获得）→ 修复的仙剑（第289章）
- 功法：基础剑法 → 天剑九式（第三式）
- 关键物品：母亲遗物（第300章才发现有秘密）

## 创作约束（强制规则）
- 说话方式：少言，但每句话都有分量。不用"呢""吧"等语气词。生气时反而更安静。
- 行为模式：保护欲强。有人伤害身边人会暴走。独处时会发呆看远方。
- 禁止：不会主动惹事，不会油嘴滑舌，不会在战斗前长篇大论
- 口头禅：无（特意设计为寡言者）

## 弧光进度
- 当前弧（觉醒弧）：78% 完成。还有3章到达本弧终点：做出"牺牲挚友还是放弃力量"的选择
```

### 4.2 人物状态更新规则

- **每5章**：小更新（如有变化）
- **每卷结束**：强制完整更新
- **人物经历重大事件**：立即更新关系/心理/能力字段
- **人物退出活跃状态**：标记为 `status: inactive`，归档到 `characters/archive/`
- **人物重新进入活跃状态**：从归档恢复，补充空白期摘要

### 4.3 人物一致性验证规则

每章创作完成后对本章出场的每个角色检查：
```
□ 说话方式与状态文件一致？
□ 行为符合弧光进度？
□ 没有违反"禁止"约束？
□ 如果有性格变化，是否符合铺垫要求？
```

## 5. 伏笔管理系统

### 5.1 伏笔文件结构

`foreshadowing.json`（JSON格式，方便程序化操作）

```json
{
  "active": [
    {
      "id": "FS-001",
      "planted_chapter": 45,
      "planted_context": "主角在拍卖会买到一把残剑",
      "nature": "物品来源之谜",
      "expected_reveal_range": [200, 300],
      "importance": "major",
      "related_characters": ["张三", "赵六"],
      "status": "active",
      "hints_laid": [45, 89, 156],
      "revealed_chapter": null
    }
  ],
  "revealed": [],
  "dropped": []
}
```

### 5.2 伏笔操作协议

- **埋入**：创作时 hook 写到章末，自动检查是否需要注册新伏笔
- **触发**：每章开始时，从 `foreshadowing.json` 筛选 `expected_reveal_range` 包含当前章的活跃伏笔 → 提示 agent 考虑回收
- **回收**：标记 `revealed_chapter`，移入 `revealed` 列表
- **丢弃**（死伏笔）：如果伏笔超过预期回收范围 200 章仍未回收，标记原因移入 `dropped`

## 7. 质量锚点机制

### 7.1 锚点分布

```
第1章 → 测试章（唯一需要用户确认的章节，确认基调和风格）
第50章 → 卷1完成，卷末审查（全面质量 + 人物弧光 + 伏笔健康度）
第100章 → 卷2完成，卷末审查
第150章 → 卷3完成，卷末审查
...
第300章 → 弧1完成，弧级审查（深度审查：弧光完成度、伏笔回收率、读者体验曲线）
第600章 → 弧2完成，弧级审查
...
第3000章 → 全书完成，终极审查
```

### 7.2 卷末审查检查项

```
1. 本卷总字数 ≥ 200,000 字（50章×4000字）
2. AI痕迹平均评分 ≥ 80%
3. 伏笔回收率 ≥ 40%（本卷新埋的伏笔至少回收40%）
4. 所有出场角色行为符合状态机
5. 节奏曲线健康（有波峰波谷，非单调）
6. 每5章至少1个爽点/反转
7. 最后3章有加速感（不能卷末拖沓）
8. 人物弧光按计划推进（偏差≤1章）
```

## 9. 创作执行流程

### 第 0 步：弧级规划（一次性）

1. 用户定义：总章数、题材、主角起点和终点、核心主题
2. agent 生成 6-10 个弧的弧级规划 `arc-plan.md`
3. 用户确认弧级规划

### 第 1 步：首卷初始化（每次启动一个新卷时执行）

1. 根据弧规划生成本卷规划 `volume-plans/vol-XX.md`（50章）
2. 初始化/更新人物状态文件
3. 初始化伏笔文件（或从上一卷继承）
4. 生成写作计划 JSON（与 novelist 兼容，50章）
5. **写完第1章后暂停，让用户确认基调**

### 第 2 步：批量创作（50章 × 逐章创作循环）

每章创作流程：

```
开始创作第X章时，上下文中装入：
  1. chapterType + 本章大纲  – 从 volume plan 提取
  2. 上章完整摘要(300字)     – 从 volume summary 提取
  3. 本章出场角色状态       – 从 character state files 提取相关角色
  4. 本章可回收的伏笔       – 从 foreshadowing.json 筛选
  5. 世界状态相关片段       – 从 world-state.md 提取
  6. 卷总结(500字)          – 从 volume summary 提取
  7. 弧总结(500字)          – 从 arc summary 提取
  8. 总览摘要(300字)        – 小说的终极目标

创作：
  9. 按 chapterType 加载技法约束
  10. 撰写 → 润色 → AI扫描 → 字数检查

收尾：
  11. 追加章节摘要到 volume summary
  12. 更新人物状态文件（如有变化）
  13. 更新世界状态快照（如有变化）
  14. 更新 foreshadowing.json（新埋伏笔 / 回收伏笔）
  15. 上下文卸载 ← 清除本章正文，只留状态索引
  16. 滚动规划: 每10章刷新后续5章大纲
  17. 继续下一章
```

### 第 3 步：卷完成仪式（每50章执行）

```
1. 生成完整的卷总结（1000-2000字）
2. 更新所有活跃人物状态文件
3. 检查弧级规划对齐度
4. 生成卷末审查报告
5. 展示给用户确认 → 决定：继续下一卷 / 调整弧级规划 / 暂停
6. 开始下一卷
```

### 第 4 步：弧完成仪式（每300-500章执行）

```
1. 生成弧1完成总结 + 读者体验曲线
2. 审计伏笔回收率
3. 深度审查人物弧光完成度
4. 用户确认后 → 下一个弧
```

## 10. 与现有系统的兼容性

### chinese-novelist 需要升级的部分

| 现有组件 | 3000章适配 | 改动 |
|----------|-----------|------|
| Phase 0 初始化 | `arc-plan.md` + `volume-plan` 替代一次性规划 | 新增弧/卷初始化 |
| Phase 1 问答 | 增加总章数、弧数、每卷章数的选项 | 扩展问答 |
| Phase 2 规划 | limit 50章/卷，生成 volume plan | 大幅修改 |
| Phase 3 创作 | 上下文装载改为"检索式"而非"全部加载" | 大幅修改 |
| Phase 4 校验 | 增加卷末审查 + 弧级审查 | 扩展 |

### chinese-writing-skills 不需要修改（技法不变）

### novel-writing-workflow 需要增加

- 卷级调度：管理卷的启动/创作/完成仪式
- 外部文件检索协议：什么时候需要读什么文件
- 上下文装载清单：每章开始前需要装载什么（精确的文件路径）

## 11. 关键数字验证

```python
# 300章后的各项累积（即使累积摘要也失控了，但我们用卷摘要替代逐章摘要）

每卷摘要: 1,500字/卷
卷数量(3000章): 60卷
卷摘要累积: 60 × 1,500 = 90,000字 ≈ 45K tokens → 仍然过大

# 修正：卷摘要只保留"活跃卷"（当前+上一卷=2卷）
活跃卷摘要: 2 × 1,500 = 3,000字 ≈ 1,500 tokens ← 可控

弧总结: 3,000字 ≈ 1,500 tokens
总览摘要: 600字 ≈ 300 tokens
人物状态(活跃角色×10人): 10 × 200字 = 2,000字 ≈ 1,000 tokens
活跃伏笔: 50条 × 30字 = 1,500字 ≈ 750 tokens
世界状态: 1,000字 ≈ 500 tokens
当前章大纲: 200字 ≈ 100 tokens
上一章摘要: 300字 ≈ 150 tokens
────────────────────────
上下文占用总计: ≈ 5,800 tokens
占上下文窗口(200K): 2.9%
```

**5,800 tokens 就能驱动 3000 章的创作——无论在写第50章还是第2999章。**

## 12. 实施优先级

| 优先级 | 功能 | 理由 |
|:---:|------|------|
| **P0** | 弧/卷/章三层规划 | 没有这个根本无法开始规划 |
| **P0** | 人物状态机文件 | 没有这个，人物在100章后必然崩塌 |
| **P0** | 伏笔管理系统 | 没有这个，跨500章的伏笔必然丢失 |
| **P0** | 上下文检索式装载协议 | 替代当前的"全量加载摘要"模式 |
| **P0** | 卷完成仪式 | 质量锚点和用户确认的唯一节点 |
| **P1** | 世界状态追踪 | 长篇世界观必然演变 |
| **P1** | 弧级审查 | 300章一次的大节点 |
| **P2** | 人物归档/恢复 | 角色暂时退场后回归时衔接 |
