## 1. 核心矛盾

```
上下文窗口：~200K tokens
3000章的摘要累积：~135K tokens（仅摘要就占 67%）
3000章的伏笔累积：~60K tokens
3000章的人物状态：~50K tokens
────────────────────────
总信息量 >> 窗口容量
```

**传统做法（逐章摘要叠加）在 300 章就爆炸**。

## 2. 分层记忆模型 v3 — 三层记忆

### 2.0 设计哲学

v2 能回答「怎么写这章」（大纲/人物/伏笔/风格），但无法回答「这段故事发展到哪了」「读者什么心情」。
v3 新增**故事级记忆层**来解决故事线跨越数百章后的连贯性问题。

```
┌──────────────────────────────────────────────────────────────────┐
│                     故事级记忆 (Story Memory)                      │
│  篇章规划 + 篇章总结 + 故事线追踪 + 读者情绪曲线 + 故事线总览       │
│  → 回答"这段故事发展到哪了/读者什么心情/线怎么交织"               │
│  → 上下文: ~1,650 tokens  │  文件: storylines/, part-plan.md, ...  │
├──────────────────────────────────────────────────────────────────┤
│                     工程级记忆 (Engineering Memory)                │
│  弧/卷/章规划 + 人物状态机 + 关系矩阵 + 伏笔 + 风格 + 世界状态      │
│  → 回答"下一章写什么/谁在/什么关系/什么伏笔/怎么写"               │
│  → 上下文: ~6,050 tokens  │  文件: arc-plan.md, volume-plans/, ...  │
├──────────────────────────────────────────────────────────────────┤
│                     会话级记忆 (Session Memory)                    │
│  上章全文 + 衔接缓存 + 创作笔记 + 预热状态                         │
│  → 回答"上一章写了什么/现在是什么语气/有什么临时决策"              │
│  → 上下文: ~4,000 tokens  │  文件: chapters/, creative-notes.md     │
└──────────────────────────────────────────────────────────────────┘
                        上下文总计: ~11,700 tokens (5.85%)
```

### 2.2 为什么需要故事级记忆？（v2 → v3 的关键洞察）

v2 系统有能力回答：
- ✅ "这一章发生了什么"（大纲）
- ✅ "这个角色现在是什么状态"（人物状态机）
- ✅ "世界现在是什么状态"（世界状态）
- ✅ "这条伏笔什么时候回收"（伏笔管理）

v2 系统无法回答：
- ❌ "这条故事线从200章前开始，现在发展到哪一步了"
- ❌ "读者在这条线上此刻应该是什么心情"
- ❌ "有哪些线同时在进行中，有没有线被遗忘了"
- ❌ "这一章和300章前的某个场景有呼应关系"

**故事级记忆就是为了解决这4个❌**。

详见 companion doc: [story-memory-system.md](story-memory-system.md)

### 上下文卸载清单 v3（每章创作后执行）

```
[每章完成后执行的文件更新 — v3 新增 storylines/ part 系列]

故事级更新（⭐v3）:
  1. 故事线进展 → 更新 storylines/{线名}.md（如有新段开始/结束）
  2. 篇章-章节映射 → 更新 part-chapter-map.json（如有新呼应关系）
  3. 读者情绪 → 更新 part-chapter-map.json 的情绪字段

工程级更新:
  4. 本章衔接缓存 → 写入 volume summary（覆盖旧的）
  5. 本章摘要 → 追加到 volume summary
  6. 人物状态变更 → 更新 characters/*-state.md
  7. 关系矩阵变更 → 更新 relation-matrix.md
  8. 新埋/回收伏笔 → 更新 foreshadowing.json
  9. 创作决策 → 追加到 creative-notes.md
  10. 世界状态变更 → 更新 world-state.md
  11. 章节索引 → 追加到 chapter-index.json
  12. 金句/名场景 → 追加到 notable-lines.json（如有）
  13. 会话日志 → 追加到 session-log.jsonl

上下文清理:
  14. 保留上章全文（不移除！→ 作为下一章的衔接感保障）
  15. 移除更早的章节正文（上下文始终只保留最近2章正文）
  16. 移除故事线详情段落（下次重新检索）
  17. 保留：卷/弧/篇章总结 + 故事线总览 + 当前章大纲
```

**关键规则**：上章全文始终保留在上下文；故事级记忆每次按需检索；上下文永不累积历史数据。

## 3. 续写预热协议（跨会话/跨卷恢复）

当 agent 在新会话中、或隔了一段时间后继续创作超长篇时，必须执行以下预热流程才能开始创作：

### 3.1 预热步骤（Cold Start SOP）

```
┌─────────────────────────────────────────────────┐
│            续写预热协议（强制执行）               │
│                                                  │
│ 1. 读 arc-plan.md — 了解「这部小说最终要去哪」   │
│ 2. 读 arc-summaries/当前弧 — 了解「目前弧的目标」 │
│ 3. 读 volume-summaries/上一卷 — 了解「最近发生了什么」│
│ 4. 读 volume-summaries/当前卷 — 了解「现在在哪」  │
│ 5. 读 creative-notes.md 最近20条 — 「之前做了哪些创作决策」│
│ 6. 读上章全文（5000字）— 衔接「文字层面的感觉」    │
│ 7. 读 relation-matrix.md — 刷新「人物关系直觉」   │
│ 8. 读 world-state.md — 刷新「世界观状态」         │
│ 9. 读 style-baseline.json — 刷新「写作风格锚点」   │
│ 10. 向用户输出 500 字预热摘要，确认理解正确        │
└─────────────────────────────────────────────────┘
```

### 3.2 预热摘要模板

```

  - FS-047：上古遗迹的门为什么对主角自动开启？

  - 王五：元婴中期，在主角身边的最后支持
```

## 4. 关系矩阵（Relationship Matrix）

### 4.1 为什么需要？

当前人物状态文件是**角色A→角色B**的单向视角。当张三-state.md 写「李四：信任」但李四-state.md 写「张三：暗中利用」时，一致性被破坏。需要一份**单一事实来源**的关系矩阵。

### 4.2 关系矩阵文件 `relation-matrix.md`

```markdown
# 人物关系矩阵
最后更新：第 347 章
活跃角色数：12

## 关系状态矩阵

| A \ B | 张三 | 李四 | 王五 | 赵六 | 天剑掌门 | 暗影 |
|-------|------|------|------|------|----------|------|
| 张三  | —    | 挚友 | 爱人 | 宿敌 | 怀疑    | 陌生↓ |
| 李四  | 挚友 | —    | 敬重 | 中立 | 信任    | 无交集 |
| 王五  | 爱人 | 敬重 | —    | 敌意 | 无交集  | 无交集 |
| 赵六  | 宿敌 | 中立 | 敌意 | —    | 利用关系| 盟友 |
| 天剑掌门| 警惕| 寄望 | 无交集| 被利用| —      | 暗中调查|
| 暗影  | 观察 | 无交集| 无交集| 合作  | 警觉    | —    |

关系类型：挚友/爱人/宿敌/信任/怀疑/尊敬/敌意/中立/利用/合作/无交集/陌生
箭头表示关系不对等方向：↑A仰视B  ↓A被B影响  →A→B单向

## 最近关系变化（最近10章）

- 第345章：张三—赵六 从「宿敌(死斗)」→「休战」
- 第347章：王五—张三 新增「订婚」状态
- 第340章：暗影 被发现也在观察 天剑掌门

## 关系强度量化（0-10）

| 关系对 | 强度 | 最近趋势 | 说明 |
|--------|:---:|:---:|------|
| 张三—李四 | 9 | → 稳 | 生死之交 |
| 张三—王五 | 8 | ↗ 升 | 刚订婚 |
| 张三—赵六 | 7(敌) | ↘ 降 | 从死斗转为休战 |
| 张三—暗影 | 2 | → 稳 | 还未知是敌是友 |
```

### 4.3 关系矩阵更新规则

- **每章结束**：检查出场角色间是否有关系变化，有则更新
- **每卷结束**：强制完整审计（逐对检查一致性）
- **新增角色**：在矩阵中增加对应行和列
- **角色退场**：保留在矩阵但标记为「退场」（保持完整性）

## 5. 创作笔记（Creative Notes）

### 5.1 为什么需要？

Agent 在创作过程中会做临时决策——比如「让王七提前3章登场」「给李四增加了一个习惯动作」「这一章用了蓝色作为人物情绪的主色调」。这些决策在会话中一闪而过，但可能影响后文几十章的创作。

### 5.2 创作笔记格式 `creative-notes.md`

```markdown
# 创作笔记
最后更新：第 347 章

## 活跃决策

### 人物决策
- [第345章] **赵六的休战不是真心的**：他休战只是为了争取时间修复内伤。预计5章后会反扑。
- [第320章] 王五开始学习阵法——会在第380章左右成为关键技能

### 情节决策
- [第340章] 将上古遗迹的入口改到南海（原本规划是北荒），因为南海的「水」元素和主角的剑法更搭
- [第300章] 父母遗物的秘密设为双层：第一层(第300章揭示)→第二层(第600章揭示)→第三层(结尾揭示)

### 风格决策
- [第100章] 从本章开始，战斗场面统一用「短句+快节奏」策略（每句≤15字），区别于日常叙述
- [第200章] 确定了主角内心的独白用括号表示，旁人的内心不写

### 待办事项
- [ ] 第350章左右需要让李四得知赵六休战的真相（关于内伤）
- [ ] 第400章前需要解释暗影为什么也在观察天剑掌门
- [ ] 王五的阵法学习进度需要展示（已过16章未提及，可能读者忘了）

## 废弃决策（历史留档，避免重复犯错）
- [第89章] ~~考虑让李四背叛~~ → 放弃，改为第203章的猜疑线，但不真背叛
```

### 5.3 创作笔记的操作规则

- **添加**：只要有影响后文的临时决策，立即添加
- **查阅**：每章开始前从预热协议中读取最近20条
- **标记完成**：待办事项完成后标记 ✅
- **不被卸载**：这是一个持久化文件，不在上下文卸载范围内
- **限制活跃条目**：保持在 30 条以内。太久远的条目降级到「历史决策」存档

## 6. 风格基线（Style Baseline）

### 6.1 为什么需要？

3000章可能跨不同 session、不同模型、甚至不同版本的 agent。风格漂移会导致小说前后像两个人写的。需要一个**可量化和可被 agent 参考的风格锚点**。

### 6.3 风格漂移检测

每完成一卷（50章），对最近5章取样和执行风格度量，如果偏离风格基线超过阈值：

```
⚠️ 风格漂移警告 (卷8, 第351-400章)：
  - 句子平均长度: 22.3 → 风格基线 28.5（偏差 -6.2，超过阈值5）
  - 对话占比: 18% → 基线32%（偏差 -14%，超过阈值8%）
  - 可能原因：战斗场景占比较高导致句子缩短+对话减少
  
建议：在下一卷的过渡章中有意恢复中长句和对话比例
```
